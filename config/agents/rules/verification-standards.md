# Verification Standards

作業着手・調査・結論付けの各段階で、根拠を確認してから前進するための横断ルール。
特定ドメインに依存しない一般則として運用する。

## キックオフ時の確認

タスク開始後、最初の編集・コマンド実行・外部書き込みのいずれかを行う前に、以下を確認する:

- **対象リポジトリ**: 現在のワーキングツリーが指示の対象と一致しているか
- **ワークツリー / ブランチ**: 想定どおりの作業ディレクトリ・ブランチ上にいるか
- **スコープ解釈**: 依頼が複数解釈できる場合、どの解釈で進めるかを一言で宣言する

候補が一意に絞れる場合は確認を求めず宣言のみで進める。複数解釈が同程度妥当なときに限り、ユーザーに選択を仰ぐ。

## 仮定の明示と一次情報での検証

時刻・タイムゾーン、命名規約、スキーマ、識別子の形式、enum 値、外部 API の単位など、**外部システムの暗黙の規約に依存する判断**をするときは:

1. 仮定をそのまま採用しない
2. 一次情報（スキーマ定義、設定ファイル、メタデータ、ドキュメント、サンプルデータ）で実体を確認する
3. 確認できない場合は、仮定を明示したうえで合意を取ってから進める

コード上の見かけと実体が乖離しやすい領域は特に注意する（型に現れない単位、暗黙のタイムゾーン、null と空文字列の区別など）。

## 否定的結論には複数ソースの整合を求める

「使われていない」「参照されていない」「存在しない」といった**否定的結論**は、単一ソースの検索結果だけでは下さない。少なくとも次の異なる種類のソースで cross-check してから断定する:

- ソースコード横断検索（リポジトリ全文・複数リポジトリ）
- 実行履歴・呼び出し履歴・ログに相当する動的記録
- 文字列結合・命名規約・テンプレートによる動的参照（静的 grep に映らない参照経路）

単一ソースで 0 件だった結果は「証拠が出ていない」状態であって「不在の証明」ではない。否定結論は cross-source の整合が取れてからにする。

## 識別子の事前確認

外部スキーマ・テーブル・カラム・API フィールド・設定キーに対してクエリ・テスト・参照コードを書く前に、識別子の存在と綴りを一次情報で確認してから実装する。「先に書いて実行時エラーで気付く」順序は避ける。
