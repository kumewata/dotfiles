# Steering コンテキスト自動取得の設計

## 概要

作業開始時に `steering-research` エージェントを使って現リポジトリの steering ドキュメント（requirements.md, design.md, tasklist.md）を自動取得し、そのナレッジを踏まえて作業を行うワークフローを steering スキルに統合する。

## 動機

steering ディレクトリ（`~/.local/state/steering/<owner>--<repo>/`）にはリポジトリの要件・設計・タスクが蓄積されているが、現状はユーザーが明示的に参照を指示しないと活用されない。兄弟リポジトリを `../other-repo` で参照してモノレポのように扱うパターンと同じ発想で、steering ディレクトリの知識を自動的に作業コンテキストに取り込む。

## 設計

### 変更1: steering スキルに「コンテキスト取得フェーズ」を追加

#### 位置

既存のモード判定（Mode 1/2/3）の**前段**に、自動コンテキスト取得フェーズを挿入する。

#### フロー

```
作業指示を受ける
  → steering スキル発動
  → Phase 0: コンテキスト取得
    → steering-research エージェントに問い合わせ:
      「現リポジトリ (<owner>--<repo>) の進行中タスク (status: in_progress or pending) を要約せよ。
       各タスクについて: タイトル、ステータス、完了率、現在のフェーズを含めること。」
    → エージェントが要約を返す（haiku の自コンテキストで検索・読み込み）
    → 要約をメインコンテキストに保持
  → 関連する steering ドキュメントがある場合:
    → それを踏まえてモード判定（既存タスクの続き → Mode 2、新規 → Mode 1）
  → steering ドキュメントがない場合:
    → 通常通り Mode 判定
```

#### コンテキスト節約の仕組み

- steering-research エージェント（haiku モデル）がサブエージェントとして検索・読み込みを行う
- メインコンテキストに載るのは**要約のみ**（数百トークン程度）
- 生の steering ドキュメント全文はメインコンテキストに載らない
- 必要に応じて個別ファイルを Read する判断はモード判定後に行う

#### steering-research への問い合わせプロンプト

```
現リポジトリ {owner}--{repo} の steering ドキュメントを検索し、以下を要約してください:

1. 進行中のタスク (status: in_progress または pending) の一覧
   - 各タスク: タイトル、ステータス、完了率、現在のフェーズ
2. 直近で完了したタスク (status: done) があれば、タイトルのみ

検索ベース: ~/.local/state/steering/{owner}--{repo}/
ドキュメントが見つからない場合は「該当なし」と報告してください。
```

### 変更2: skill-triggers ルールへの追記

`config/agents/rules/skill-triggers.md` の steering セクションに、コンテキスト自動取得が発動することを明記する。

#### 追記内容

steering の必須トリガーセクションに以下を追加:

```markdown
**コンテキスト自動取得**: steering スキル発動時、モード判定の前に
steering-research エージェントを使って現リポジトリの進行中タスクを自動取得する。
これにより過去の要件・設計がコンテキストに反映される。
```

### 変更しないもの

- `steering-research` エージェント定義 — 既に必要な検索機能を持っている
- steering スキルの Mode 1/2/3 のロジック — そのまま維持
- steering テンプレート — 変更不要

## 具体的な変更箇所

### `config/agents/skills/steering/SKILL.md`

1. 「スキルの目的」セクションに「作業開始時の自動コンテキスト取得」を追加
2. 「使用タイミング」セクションに「0. 作業開始時: 既存 steering ドキュメントのコンテキスト取得」を追加
3. Mode 1 の手順1（リポジトリ名の自動検出）の前に「Phase 0: コンテキスト取得」セクションを新設

### `config/agents/rules/skill-triggers.md`

1. steering セクションにコンテキスト自動取得の説明を追記

## トレードオフ

| 項目          | メリット                                           | リスク                                      |
| ------------- | -------------------------------------------------- | ------------------------------------------- |
| 自動取得      | ユーザーが意識せずとも過去の要件・設計が反映される | 毎回 steering-research が走るオーバーヘッド |
| haiku 委譲    | メインコンテキスト消費が最小限                     | haiku の要約精度に依存                      |
| steering 統合 | シームレスなワークフロー                           | steering スキルの複雑度が微増               |

## リスク軽減

- steering ドキュメントがないリポジトリでは steering-research が「該当なし」を返すだけで、オーバーヘッドは最小
- haiku の要約が不十分な場合、ユーザーが直接 Read で詳細を確認できる（既存の Mode 2 フロー）
